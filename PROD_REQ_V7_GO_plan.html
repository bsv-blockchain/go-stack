Developer plan for the V7 Go wallet schema migration

This standalone implementation guide lays out the move from the current Go schema to a cleaner V7 model: canonical network transaction state in one table, per-user actions in another, and a single optimized outputs table that preserves the hot spendable-output query path.

1 canonical transaction row per txid for proof, broadcast, and processing state.
N per-user action rows for labels, descriptions, notifications, and visibility.
0 joins on the common spendable outputs query after migration.
30d recommended rollback window before dropping legacy tables.

Current Go Schema

The migration starts from the schema that exists today. V7 is intentionally scoped to this Go implementation and does not depend on earlier V2-V6 drafts.

Per-user

transactions

Currently holds the user-facing lifecycle: status, description, labels, satoshis delta, references, and notification fields.

Shared state

known_txs

Closest existing canonical transaction record. Stores txid processing, proofs, broadcast attempts, idempotency keys, and raw transactions.

Hot path

outputs + user_utxos

Outputs hold raw output data while user_utxos acts as the fast per-user index for spendable-output queries.

transactions User lifecycle and display metadata
->
known_txs Network processing, proof, broadcast state
->
outputs + user_utxos Raw outputs plus user-specific spendable index

Tables that remain conceptually unchanged

output_baskets, output_tags, output_tags_map, tx_labels, tx_labels_map, and related supporting tables continue to exist. V7 changes where transaction labels point: tx_labels_map targets actionId instead of the old transactionId.

V7 Target Schema

The target model separates global txid truth from per-user action state while keeping the output lookup path denormalized for speed.

Hot spendable-output query

This remains a single-table index scan. No joins are introduced on the common wallet path.

SELECT * FROM outputs
WHERE user_id = ?
  AND spendable = true
  AND basketId = default_basket
ORDER BY satoshis DESC
LIMIT 100;

Supporting tables

chain_tip stores the current chain height singleton. tx_audit records event history at transactionId and actionId scope. monitor_lease coordinates background monitoring work.

Migration Guide

Run this once, non-destructively first, then cut over atomically after verification. Reads can continue until the brief final rename window.

01

Create new tables

Create actions, transactions_new, and outputs_new without touching current production tables.

CREATE TABLE actions ( ... full definition as in target actions ... );
CREATE TABLE transactions_new ( ... full definition as in target transactions ... );
CREATE TABLE outputs_new ( ... full definition as in target outputs ... );
02

Backfill actions from current transactions

Preserve the per-user lifecycle and convert nosend status into user_nosend.

INSERT INTO actions (
  userId, transactionId, reference, description, is_outgoing,
  satoshis_delta, user_nosend, hidden, user_aborted,
  notify_json, created_at, updated_at
)
SELECT
  userId, transactionId, reference, description, isOutgoing,
  satoshis_delta,
  CASE WHEN status = 'nosend' THEN TRUE ELSE FALSE END,
  FALSE, FALSE, notify_json, created_at, updated_at
FROM transactions;
03

Migrate known_txs into canonical transactions

Create or update each transactions_new row with processing state, proof fields, broadcast counters, idempotency, and raw transaction data.

sending -> sending unmined -> sent completed -> proven reorg -> reorging invalidTx -> invalid
04

Migrate outputs

Preserve the fast path by materializing user-specific output rows directly into outputs_new.

INSERT INTO outputs_new (
  txid, vout, user_id, amount, locking_script, spendable,
  basketId, label, description, is_coinbase, matures_at_height,
  spent_by, created_at, updated_at
)
SELECT
  o.txid, o.vout, u.userId, o.satoshis, o.lockingScript,
  u.spendable, o.basketId, o.label, o.description,
  o.is_coinbase, o.matures_at_height, o.spentBy,
  o.created_at, o.updated_at
FROM outputs o
JOIN user_utxos u ON u.outputId = o.outputId;
05

Re-point foreign keys

Update tx_labels_map so per-user transaction labels point to actionId. Update any other references that target the old per-user transaction record.

06

Atomic cutover

Rename old tables aside and promote the V7 tables inside one transaction or a brief maintenance window.

BEGIN;
  RENAME TABLE transactions TO transactions_old;
  RENAME TABLE transactions_new TO transactions;
  RENAME TABLE outputs TO outputs_old;
  RENAME TABLE outputs_new TO outputs;
  RENAME TABLE user_utxos TO user_utxos_old;
COMMIT;
07

Post-migration cleanup

Drop known_txs, user_utxos_old, and transactions_old only after verification. Rebuild old indexes on the new tables and run the full conformance suite.

Spendability Derivation

The boolean remains cached on outputs and is refreshed atomically whenever transaction state, spend state, script availability, coinbase maturity, or chain tip changes.

Cached spendable expression

spendable = (
  transactions.processing IN (
    'sent', 'seen', 'seen_multi', 'unconfirmed', 'proven'
  )
)
AND outputs.spent_by IS NULL
AND outputs.locking_script IS NOT NULL
AND (
  NOT outputs.is_coinbase
  OR outputs.matures_at_height <= chain_tip.height
)
queued sending sent seen seen_multi unconfirmed proven reorging invalid doubleSpend unfail frozen nosend nonfinal

Conformance Tests

Use this checklist while implementing. Completion is stored locally in the browser for the developer using this page.

Invariant

Canonical tx state

Broadcast, proof, blockchain status, raw transaction, and provider status exist exactly once per txid.

Invariant

User scope

Descriptions, labels, visibility, abort state, and notifications are scoped to actions.

Risk

Index parity

Rebuild the indexes required by the old hot paths before declaring the cutover stable.

Rollout Recommendation

Keep the release operationally conservative: stage with production-like data, verify query performance, then retain rollback tables for one release.
